</Info>

## Problem

When using `LocalDockerRuntime`, the Dockerfile's `CMD` defines the container's entry point — this is what actually starts the agent process. The agent template also has a `spawn` field, which is used by local runtimes to start the agent.

Today, when a Docker runtime is used, the `spawn` field is silently ignored and the container's `CMD` takes precedence. This creates a confusing situation:

- A user defines `spawn: { command: "node", args: ["agent.js"] }` on a template
- The Dockerfile has `CMD ["python", "agent.py"]`
- The template's `spawn` is silently ignored
- The user expects `node agent.js` but gets `python agent.py`

There's no warning, error, or indication that the two conflict. This leads to debugging sessions that could be avoided.

## Proposed solution: build-time validation

When `LocalDockerRuntime` builds or inspects a Docker image, it should extract the `CMD` from the image metadata and compare it against the template's `spawn` field. If they differ, Flamecast should warn or error depending on the configuration.

### Detection

After building or pulling the Docker image, Flamecast inspects the image config to extract the `CMD`:

```typescript
// Pseudocode
const imageConfig = await docker.inspectImage(imageId);
const dockerCmd = imageConfig.Config.Cmd; // e.g. ["python", "agent.py"]
const templateSpawn = template.spawn;     // e.g. { command: "node", args: ["agent.js"] }

if (dockerCmd && templateSpawn && !commandsMatch(dockerCmd, templateSpawn)) {
  // Conflict detected
}
```

### Behavior modes

The validation behavior should be configurable on the `LocalDockerRuntime`:

```typescript
const flamecast = new Flamecast({
  runtimes: {
    docker: new LocalDockerRuntime({
      spawnConflict: "warn", // "warn" | "error" | "ignore"
    }),
  },
});
```

| Mode | Behavior |
|---|---|
| `"warn"` | Log a warning at session creation time. The session proceeds using the Dockerfile `CMD`. This is the default |
| `"error"` | Reject the session with a descriptive error message explaining the conflict |
| `"ignore"` | No validation. Current behavior |

### Warning message

When a conflict is detected in `"warn"` mode:

```
[flamecast] Warning: Template "my-agent" has spawn command "node agent.js" 
but the Docker image CMD is "python agent.py". The Docker CMD takes precedence 
when using LocalDockerRuntime. To suppress this warning, either:
  - Remove the spawn field from the template
  - Update the Dockerfile CMD to match
  - Set spawnConflict: "ignore" on the runtime
```

### Error message

In `"error"` mode, the session creation fails with:

```json
{
  "error": "spawn_conflict",
  "message": "Template \"my-agent\" spawn command \"node agent.js\" conflicts with Docker image CMD \"python agent.py\". The Docker CMD takes precedence when using LocalDockerRuntime.",
  "templateSpawn": ["node", "agent.js"],
  "dockerCmd": ["python", "agent.py"]
}
```

## Edge cases

### No CMD in Dockerfile

If the Dockerfile has no `CMD` (or uses `ENTRYPOINT` only), validation is skipped. The `spawn` field could potentially be passed as the container's command arguments — this is a separate feature decision.

### No spawn on template

If the template omits `spawn`, there's no conflict to detect. This is the recommended pattern for Docker-only templates:

```typescript
{
  id: "my-agent-docker",
  name: "My agent (Docker)",
  runtime: "docker",
  // No spawn — the Dockerfile CMD handles it
}
```

### CMD with shell form

Docker CMD can be specified in shell form (`CMD node agent.js`) or exec form (`CMD ["node", "agent.js"]`). The validation should normalize both forms before comparison. Shell form wraps the command in `/bin/sh -c`, so:

- `CMD node agent.js` → `["/bin/sh", "-c", "node agent.js"]`
- `CMD ["node", "agent.js"]` → `["node", "agent.js"]`

The comparison should check the underlying command, not the shell wrapper.

### Setup scripts and CMD

If a template has both `setup` (a shell script string — see [setup scripts RFC](/rfcs/setup-scripts)) and a Docker `CMD`, the lifecycle is:

1. Container starts with `CMD`
2. `setup` runs inside the container via `sh -c` (if defined)
3. The `CMD` process is already running

This means `setup` runs alongside the `CMD` process, not before it. If the template needs setup to run first, the Dockerfile should use a custom entrypoint that waits for a signal before starting the agent.

*Proposed:* For Docker runtimes, change the lifecycle so the container starts with a wait loop, `setup` runs first, then the agent process starts. This avoids the race condition but requires Flamecast to control the container entrypoint.

## Open questions

1. **Override mode.** Should there be a mode where the template's `spawn` overrides the Docker `CMD`? For example, pass the spawn command as `docker run ... <spawn command>` instead of relying on `CMD`. This would make Docker templates work like local templates with the same spawn field.

2. **Remote images.** When using pre-built images (not built from a local Dockerfile), should Flamecast pull and inspect the image at registration time or defer validation to session creation?

3. **ENTRYPOINT interaction.** How should validation handle `ENTRYPOINT` + `CMD` combinations? If `ENTRYPOINT` is set, `CMD` provides default arguments to it, which changes the semantics.
