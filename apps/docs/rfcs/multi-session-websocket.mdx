## Background: ACP sessions

In the ACP protocol, a single agent process can host multiple concurrent chat sessions. When Flamecast starts an agent, it calls `newSession()` on the ACP connection to create a conversation context. Each session gets its own `sessionId`, prompt queue, permission state, and event history — but they all share the same underlying agent process and workspace.

## Problem

Today, the WebSocket endpoint delivers every event for a session over a single connection:

```
ws://localhost:3001/ws/sessions/:sessionId
```

This is simple but inflexible:

- **No channel filtering.** A UI that only needs terminal output still receives every conversation event, permission request, and file system snapshot. A dashboard that only cares about session lifecycle gets flooded with message chunks
- **One session per connection.** A tabbed editor with 10 conversations needs 10 WebSocket connections, each with its own handshake, heartbeat, and reconnection logic
- **No cross-session visibility.** There's no way to subscribe to file system changes across all sessions in an agent, or to terminal output from a specific session without also receiving its conversation events
- **All or nothing.** Clients can't opt into the event types they care about

## Proposed solution: WebSocket adapter with channel subscriptions

Replace the flat event stream with a channel-based subscription model. Clients connect to a single WebSocket endpoint and subscribe to specific channels at different granularity levels.

### Endpoint

```
ws://localhost:3001/ws
```

One connection. Clients subscribe to the channels they need.

### Channels

A channel is a scoped event stream. Channels are hierarchical — subscribing to a broader channel includes events from its children.

| Channel | Events delivered |
|---|---|
| `agent:{agentId}` | All events for all sessions in an agent (lifecycle, conversations, terminals, file system, queue) |
| `session:{sessionId}` | All events for a specific session (conversation, permissions, queue) |
| `session:{sessionId}:terminal` | Terminal output for a session |
| `session:{sessionId}:terminal:{terminalId}` | Output from a specific terminal |
| `session:{sessionId}:queue` | Queue state changes for a session |
| `session:{sessionId}:fs` | File system snapshots for a session |
| `agent:{agentId}:fs` | File system changes across all sessions in an agent |
| `agents` | Lifecycle events for all agents (created, terminated) |

### Client-to-server messages

| Action | Fields | Description |
|---|---|---|
| `subscribe` | `channel` | Start receiving events on a channel |
| `unsubscribe` | `channel` | Stop receiving events on a channel |
| `prompt` | `sessionId`, `text` | Send a prompt to a session |
| `permission.respond` | `sessionId`, `requestId`, `body` | Respond to a permission request |
| `cancel` | `sessionId`, `queueId` | Cancel a queued prompt |
| `terminate` | `sessionId` | Terminate a session |
| `session.create` | `agentId` | Create a new session on an agent |
| `terminal.input` | `terminalId`, `data` | Send input to a terminal |
| `terminal.resize` | `terminalId`, `cols`, `rows` | Resize a terminal |
| `queue.reorder` | `sessionId`, `order` | Reorder the prompt queue |
| `queue.clear` | `sessionId` | Clear the prompt queue |
| `queue.pause` | `sessionId` | Pause queue processing |
| `queue.resume` | `sessionId` | Resume queue processing |
| `ping` | — | Heartbeat |

### Server-to-client messages

Every message includes the `channel` it belongs to and the relevant IDs:

```json
{
  "type": "event",
  "channel": "session:session-456",
  "sessionId": "session-456",
  "agentId": "agent-123",
  "event": {
    "type": "session_update",
    "data": { ... }
  }
}
```

| Type | Description |
|---|---|
| `connected` | Connection established |
| `subscribed` | Successfully subscribed to a channel. Followed by history replay for session channels |
| `unsubscribed` | Unsubscribed from a channel |
| `event` | An event on a subscribed channel |
| `session.created` | A new session was created |
| `session.terminated` | A session was terminated |
| `error` | Error message |

### Example: subscribe to specific channels

```typescript
const ws = new WebSocket("ws://localhost:3001/ws");

ws.onopen = () => {
  // Subscribe to conversation events for one session
  ws.send(JSON.stringify({ action: "subscribe", channel: "session:session-1" }));

  // Subscribe to terminal output for another session
  ws.send(JSON.stringify({ action: "subscribe", channel: "session:session-2:terminal" }));

  // Subscribe to file system changes across all sessions in an agent
  ws.send(JSON.stringify({ action: "subscribe", channel: "agent:agent-123:fs" }));
};

ws.onmessage = (msg) => {
  const data = JSON.parse(msg.data);
  console.log(`[${data.channel}]`, data.type, data.event?.type);
};
```

### Example: dashboard subscribing to agent lifecycle

```typescript
ws.onopen = () => {
  // Only receive agent/session lifecycle events — no conversation content
  ws.send(JSON.stringify({ action: "subscribe", channel: "agents" }));
};

ws.onmessage = (msg) => {
  const data = JSON.parse(msg.data);

  if (data.type === "session.created") {
    // A new session started — subscribe to its queue for status tracking
    ws.send(JSON.stringify({
      action: "subscribe",
      channel: `session:${data.sessionId}:queue`,
    }));
  }
};
```

### Example: full agent view with all channels

```typescript
ws.onopen = () => {
  // Subscribe to everything for an agent — conversations, terminals, fs, queues
  ws.send(JSON.stringify({ action: "subscribe", channel: "agent:agent-123" }));
};
```

### Creating sessions over WebSocket

```typescript
ws.send(JSON.stringify({
  action: "session.create",
  agentId: "agent-123",
}));

// Server responds with:
// { "type": "session.created", "sessionId": "session-789", "agentId": "agent-123" }
// Client is automatically subscribed to "session:session-789"
```

## REST endpoint for multi-session

To support multiple sessions per agent, add:

```
POST /api/agents/:agentId/sessions
```

**Request body**

```json
{
  "cwd": "/path/to/project"
}
```

**Response**

```json
{
  "sessionId": "session-456",
  "agentId": "agent-123"
}
```

The agent process calls `newSession()` on the ACP connection. The session shares the agent's runtime and workspace but has independent conversation state and prompt queue.

## React hooks

### `useFlamecast` — single connection, selective subscriptions

```typescript
import { useFlamecast } from "@flamecast/sdk/client/hooks/use-flamecast";

function App() {
  const flamecast = useFlamecast({ url: "ws://localhost:3001/ws" });

  return (
    <FlamecastProvider value={flamecast}>
      <AgentView agentId="agent-123" />
    </FlamecastProvider>
  );
}
```

The `useFlamecast` hook manages a single WebSocket connection. All other hooks use it as a shared transport.

### `useSession` — subscribe to a session's conversation

```typescript
import { useSession } from "@flamecast/sdk/client/hooks/use-session";

function ConversationView({ sessionId }: { sessionId: string }) {
  const {
    events,             // SessionLog[] — conversation events only
    connectionState,
    prompt,
    respondToPermission,
    cancel,
    terminate,
  } = useSession(sessionId);

  // ...
}
```

Internally subscribes to `session:{sessionId}`.

### `useTerminal` — subscribe to terminal output

```typescript
import { useTerminal } from "@flamecast/sdk/client/hooks/use-terminal";

function TerminalPanel({ sessionId }: { sessionId: string }) {
  const {
    terminals,       // TerminalSession[]
    activeTerminal,  // TerminalSession | null
    sendInput,       // (terminalId: string, data: string) => void
    resize,          // (terminalId: string, cols: number, rows: number) => void
  } = useTerminal(sessionId);

  // ...
}
```

Internally subscribes to `session:{sessionId}:terminal`.

### `useQueue` — subscribe to queue state

```typescript
import { useQueue } from "@flamecast/sdk/client/hooks/use-queue";

function QueuePanel({ sessionId }: { sessionId: string }) {
  const {
    items,       // QueueItem[]
    processing,  // QueueItem | null
    paused,      // boolean
    cancel,
    clear,
    reorder,
    pause,
    resume,
  } = useQueue(sessionId);

  // ...
}
```

Internally subscribes to `session:{sessionId}:queue`.

### `useFileSystem` — subscribe to file system changes

```typescript
import { useFileSystem } from "@flamecast/sdk/client/hooks/use-file-system";

function FileExplorer({ sessionId }: { sessionId: string }) {
  const {
    files,            // FileSystemSnapshot
    requestPreview,   // (path: string) => Promise<FilePreview>
  } = useFileSystem(sessionId);

  // ...
}

// Or subscribe to file changes across an entire agent
function AgentFiles({ agentId }: { agentId: string }) {
  const { files } = useFileSystem({ agentId });
  // Internally subscribes to agent:{agentId}:fs
}
```

### `useAgent` — subscribe to all sessions for an agent

```typescript
import { useAgent } from "@flamecast/sdk/client/hooks/use-agent";

function AgentView({ agentId }: { agentId: string }) {
  const {
    sessions,        // Map<string, SessionState>
    createSession,   // () => Promise<string>
    prompt,          // (sessionId: string, text: string) => void
    respondToPermission,
    terminate,
  } = useAgent(agentId);

  return (
    <div>
      <button onClick={() => createSession()}>New conversation</button>
      {[...sessions.entries()].map(([id, session]) => (
        <ConversationTab key={id} sessionId={id} events={session.events} />
      ))}
    </div>
  );
}
```

Internally subscribes to `agent:{agentId}`.

### Composing hooks

Each hook subscribes only to the channels it needs. A full session UI composes them:

```typescript
function SessionView({ sessionId }: { sessionId: string }) {
  const session = useSession(sessionId);    // session:{sessionId}
  const terminal = useTerminal(sessionId);  // session:{sessionId}:terminal
  const queue = useQueue(sessionId);        // session:{sessionId}:queue
  const fs = useFileSystem(sessionId);      // session:{sessionId}:fs

  return (
    <div className="session-layout">
      <Conversation events={session.events} onPrompt={session.prompt} />
      <TerminalPanel terminals={terminal.terminals} />
      <QueuePanel items={queue.items} processing={queue.processing} />
      <FileExplorer files={fs.files} />
    </div>
  );
}
```

All four hooks share the same underlying WebSocket via `FlamecastProvider`. Four channel subscriptions, one connection.

## Backward compatibility

The per-session endpoint (`ws://localhost:3001/ws/sessions/:sessionId`) continues to work unchanged — it delivers all events for that session with no subscription management. The adapter endpoint is additive.

| Endpoint | Use case |
|---|---|
| `ws://.../ws/sessions/:id` | Simple single-session UIs. No subscribe/unsubscribe needed |
| `ws://.../ws` | Everything else: multi-session, filtered channels, dashboards |

## History replay

When subscribing to a session-level channel (`session:{id}` or `agent:{id}`), the server replays relevant event history. Channel-specific subscriptions only replay events for that channel:

- `session:abc` → full conversation history
- `session:abc:terminal` → buffered terminal output only
- `session:abc:queue` → current queue state snapshot
- `session:abc:fs` → latest file system snapshot

## Connection limits

| Setting | Default | Description |
|---|---|---|
| `maxSubscriptionsPerConnection` | `100` | Maximum channel subscriptions per WebSocket |
| `maxHistoryReplayBatch` | `1000` | Maximum events replayed per channel on subscribe |

## Open questions

1. **Channel wildcards.** Should channels support glob patterns? For example, `session:*:terminal` to subscribe to terminal output from all sessions without subscribing to conversation events.

2. **Reconnection.** On reconnect, should the client re-send `subscribe` messages (stateless), or should the server remember subscriptions (stateful)? Stateless is simpler but clients may miss events during the gap.

3. **Subscription deduplication.** If a client subscribes to both `agent:123` and `session:456` (where session-456 belongs to agent-123), should they receive duplicate events? Proposed: deduplicate — each event is delivered at most once per connection.

4. **Rate limiting per channel.** High-frequency channels like terminal output could overwhelm slow clients. Should there be per-channel backpressure or coalescing?

5. **Session isolation.** Should there be a way to create isolated sessions that don't share the file system? For example, each session gets its own worktree via a `setup` script.
