Flamecast is configured through TypeScript via the `Flamecast` constructor. There are no JSON config files.

The preferred Node pattern is to export a configured `Flamecast` instance from `flamecast.config.ts` and import that same instance from your app bootstrap:

```typescript
import dotenv from "dotenv";
import { Flamecast, NodeRuntime } from "@flamecast/sdk";
import { createPsqlDatabase } from "@flamecast/storage-psql";

dotenv.config();
const db = createPsqlDatabase({
  url: process.env.DATABASE_URL,
});

export default new Flamecast({
  backend: db,
  runtimes: {
    default: new NodeRuntime(),
  },
});
```

## Constructor options

```typescript
import { Flamecast, NodeRuntime, listen } from "@flamecast/sdk";
import { DockerRuntime } from "@flamecast/runtime-docker";
import { E2BRuntime } from "@flamecast/runtime-e2b";
import { createPsqlDatabase } from "@flamecast/storage-psql";

const flamecast = new Flamecast({
  backend: createPsqlDatabase({
    url: process.env.DATABASE_URL,
  }),
  runtimes: {
    default: new NodeRuntime(),
    docker: new DockerRuntime(),
    e2b: new E2BRuntime({ apiKey: process.env.E2B_API_KEY }),
  },
  agentTemplates: [
    {
      id: "my-agent",
      name: "My agent",
      spawn: { command: "node", args: ["agent.js"] },
      setup: "npm install",
      runtime: "e2b",
    },
  ],
});

await flamecast.init();
listen(flamecast, 3001);
```

| Option | Type | Default | Description |
|---|---|---|---|
| `storage` | `FlamecastStorage` | Optional | Ready-to-use persistence backend |
| `backend` | `FlamecastBackend` | Optional | Lazy SQL backend handle. Use `createPsqlDatabase()` when you want one exported app instance plus CLI migration support |
| `runtimes` | `Record<string, Runtime>` | Required | Named runtime instances. Agent templates reference these by key |
| `agentTemplates` | `AgentTemplate[]` | Built-in templates | Initial template list. Replaces defaults when provided |
| `callbackUrl` | `string` | Auto-detected | Override the callback URL session hosts use to reach the control plane |
| `webhooks` | `WebhookConfig[]` | `[]` | Global webhook destinations merged into every session |

Provide exactly one of `storage` or `backend`. When using `backend`, call `await flamecast.init()` during startup if you want failures such as pending migrations to surface before the server begins accepting requests.

### Runtimes

Each key in the `runtimes` object becomes a named runtime that agent templates can reference. For example, if you register `e2b: new E2BRuntime(...)`, templates can use `runtime: "e2b"`.

Templates registered via `POST /api/agent-templates` can reference any runtime available on the server:

```bash
curl -X POST http://localhost:3001/api/agent-templates \
  -H "Content-Type: application/json" \
  -d '{
    "name": "E2B agent",
    "spawn": { "command": "node", "args": ["agent.js"] },
    "runtime": "e2b"
  }'
```

See [Orchestrate cloud agents](/guides/cloud-agents) for details on built-in and custom runtimes.

## Fetch handler

For serverless or custom HTTP setups, use the `fetch` handler:

```typescript
import { Flamecast, NodeRuntime } from "@flamecast/sdk";
import { createPsqlStorage } from "@flamecast/storage-psql";

const flamecast = new Flamecast({
  storage: await createPsqlStorage({ url: process.env.DATABASE_URL! }),
  runtimes: {
    default: new NodeRuntime(),
  },
});

export default flamecast.fetch;
```

## Environment variables

| Variable | Description |
|---|---|
| `FLAMECAST_PGLITE_DIR` | Override PGLite data directory (default: `<cwd>/.flamecast/pglite`) |
| `DATABASE_URL` / `POSTGRES_URL` | Postgres connection string, usually read inside `flamecast.config.ts` or your app bootstrap |
| `FLAMECAST_PORT` or `PORT` | Override the default listen port (default: `3001`) |

## Current limitations

- No authentication or multi-tenancy built in. Add your own auth layer before exposing to the internet.
- Single-process control plane. No distributed coordination across multiple instances.
- Worker deployment (e.g. Cloudflare Workers) requires custom runtime providers — `local` and `docker` providers are not available.
- Runtime reconnection across process restarts is not implemented yet.
